**AIBOUX Shop Phase 14 厳格レビュー（Grok Build視点）**

対象は主に画像SEO自動化とフィード同期基盤の強化。全体として「自動化を押し進めすぎて、信頼性と将来拡張性を犠牲にしている」状態です。以下、指定観点ごとに重大度を付けて評価します。

### 1. 出店者にSEO設定を強要せず、自動最適化が安心して伝わるUXか

**評価: 中程度の問題あり（重大度: 中）**

**指摘:**
- 自動最適化自体は強要していない点は正しい。ただし、UXとして「AIがGoogle/Bing向けに最適化しました」というメッセージをアップロード直後に表示しているのは、心理的安全性を損なっている。
- 出店者からすると「自分の画像が勝手にリネームされ、勝手にメタデータが付与された」という印象を与えやすい。特に`buildShopImageSeoKey`がかなり積極的に文脈を読み取ってslugを生成しているため、「自分の意図しない名前になった」ケースが発生しやすい。
- 現在は「最適化された」という事実しか伝えていない。**何がどう変わったのか**（例: 元のファイル名→新しいファイル名、どのキーワードが抽出されたか）が一切表示されていない。

**修正提案:**
- アップロード完了時に「元のファイル名: `xxx.jpg` → 最適化後: `black-leather-wallet-front.jpg`」のように、**変更前後の比較**を明示する。
- 「この最適化は後から無効化できます（将来機能）」という逃げ道を軽くでも提示する。
- メッセージを「AIが自動で最適化しました」から「検索エンジン向けに画像名を自動調整しました（任意で後から変更可能）」程度にトーンを抑える。

### 2. R2画像キーのSEO自動リネームが過剰・危険でないか

**評価: 危険性が高い（重大度: 高）**

**指摘:**
- `buildShopImageSeoKey`の実装が過剰に積極的。日本語辞書変換 + 複数ソースからのトークン抽出 + 除外処理まで行っており、**意図しないリネームが頻発する設計**になっている。
- 特に危険なのは以下の点:
  - 同じ商品の画像を2回アップロードすると、ほぼ確実に異なるキーが生成される（タイムスタンプ + UUIDが入っているため）。
  - `productTitle`や`keywords`が空の場合でも、元のファイル名からかなり積極的にslugを生成しようとする。
  - R2のカスタムメタデータに`seoFileName`や`seoSlug`を保存しているが、これを後から修正する手段が一切存在しない。
- 画像処理フロー（`process-image`）が存在する前提で設計されているようだが、**リネームの副作用に対する防御が極めて弱い**。

**修正提案:**
- リネームは「強く推奨」止まりにし、**元のファイル名を尊重するモード**をデフォルトにするべき。
- 少なくとも「同じJANコード + 同じpurpose」の画像については、一定期間内は同一スラッグを維持するロジックが必要。
- 現在のような「毎回ほぼ確実に新しいキーになる」設計は、将来的に画像の差し替え・バージョン管理を極めて困難にする。

### 3. JSON-LD Product/OfferがGoogle/Bingのクローラに読める構造か

**評価: 基本は通るが、甘い（重大度: 中）**

**指摘:**
- 構造自体は大きく破綻していない。ただし、以下の点でGoogleの厳格な要件に対して不十分:
  - `offers`に`seller`が一切入っていない（Googleは強く推奨）。
  - `availability`と`itemCondition`はURL形式で正しいが、`price`が数値で出力されているのは問題ないものの、**在庫状況が常に`InStock`固定**なのは危険。
  - `image`配列にalt情報が一切紐づいていない（`ai_alt_texts_json`は別管理）。
  - `gtin`としてJANコードを入れているのは良いが、`sku`と`gtin`の使い分けが曖昧。

**修正提案:**
- `offers`に最低限`seller`（`@type: Organization`）を追加。
- 将来的に在庫状態を反映できるように、`availability`を動的にできる設計にしておく。
- 画像とaltテキストの対応関係をJSON-LD上でも表現できるようにする（`image`をオブジェクト配列にするか、別途`associatedMedia`を使う）。

### 4. Cloudflare Queuesのproducer/consumer基盤が次フェーズのGoogle/Bing同期に耐えるか

**評価: 現時点でほぼ耐えられない（重大度: 高）**

**指摘:**
- これが現時点で最も深刻な問題。
- `shopFeedSyncQueue.ts`はただキューに投げているだけ。
- `worker.ts`の`queue`ハンドラは**メッセージを受け取ったら即`ack()`しているだけ**で、実際の処理は一切行われていない。
- エラーハンドリング、再試行、DLQ（Dead Letter Queue）、バッチ処理の考慮が完全に欠落している。
- 次フェーズでGoogle Merchant Center / Bing Merchant Centerへの同期を実装しようとした場合、現状の基盤では**ほぼ確実に詰む**。

**修正提案:**
- 現時点でconsumerは「受け取ったことを記録する」以上のことをすべきではないが、少なくとも以下の基盤は整えておく必要がある:
  - メッセージの`id`と処理結果をDBに永続化
  - 失敗時の再試行回数管理
  - 処理失敗時のDLQへの移動（または明示的な`retry()`）
- 現在のような「受け取ったら即ack」の実装は、**本番で絶対に使ってはいけない**。

### 総合評価と承認可否

| 観点 | 重大度 | 評価 |
|------|--------|------|
| 1. SEO強要の有無とUX | 中 | 自動化の方向性は正しいが、透明性が不足 |
| 2. R2画像キーの自動リネーム | 高 | 過剰で危険 |
| 3. JSON-LD構造 | 中 | 最低限は通るが不十分 |
| 4. Queues基盤 | 高 | 次フェーズに耐えられない |

**承認可否: 現時点では承認不可（条件付きで再レビュー必要）**

特に**観点2と観点4**が致命的です。

- 画像の自動リネームは「便利さ」より「後からの修正可能性」と「意図の尊重」を優先すべき段階にあります。
- Queuesは「次フェーズで本気で使うつもりなら、今のうちに最低限の信頼性基盤を整えておく」必要があります。

Phase 14としては「自動化をやりすぎた」状態に陥っています。自動化の恩恵を享受しつつも、**出店者と運用者が後から介入できる余地**を意図的に残す設計に修正することを強く推奨します。
